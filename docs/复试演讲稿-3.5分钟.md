## 📌 开场介绍（30秒）

各位老师好，我今天展示的项目是一个**基于 Spring Boot 的本地缓存框架**。

这个项目的核心价值在于：
- **从零实现**了支持 LRU、LFU、FIFO 三种淘汰策略的缓存系统
- 实现了 **TTL 过期机制**、**并发安全控制**和**实时监控统计**
- 应用了**策略模式、模板方法模式、门面模式**三种设计模式
- 提供了可视化 Dashboard 和 REST API 接口

技术栈：Java 17 + Spring Boot 3.3 + Thymeleaf + ConcurrentHashMap + ReentrantLock

---

## 🏗️ 系统架构（60秒）

**[展示 01-architecture.html]**

整个系统采用**五层架构设计**：

### 1. Client 层
- 浏览器 Dashboard：Thymeleaf 服务端渲染的监控面板
- REST API Client：支持缓存 CRUD、策略切换、统计查询

### 2. Web 层
- **DashboardController**：处理页面路由和表单提交
- **CacheRestController**：提供 RESTful API 接口

### 3. Service 层（业务编排中枢）
- **CacheManagerService**：协调缓存与数据源交互，判断数据来源（HIT/MISS/EXPIRED）
- **ProductRepository**：模拟数据库的慢查询（600ms 延迟）

### 4. Cache Core 层（核心实现）
- **LocalCache**：缓存引擎，负责数据的存取、TTL 判断、容量管理和统计
- 内部使用 `ConcurrentHashMap` 存储条目，`ReentrantLock` 保障复合操作的原子性
- **CacheEntry**：缓存条目，记录 key、value、创建时间、过期时间、访问次数等元数据
- **CacheEntryView**：不可变的只读快照，安全暴露给外部使用
- **CacheStats**：命中次数、未命中次数、命中率、淘汰次数、过期次数的统计快照

### 5. Eviction Policy 层（策略实现）
- **EvictionPolicy 接口**：定义 `onPut`、`onAccess`、`onRemove`、`evictKey` 四个回调
- **AbstractLinkedPolicy**：LRU 和 FIFO 共享的双向链表骨架，用哨兵节点简化边界处理
  - **LRU**：每次访问将节点移到链表尾部，淘汰头部节点（最久未访问）
  - **FIFO**：访问不改变顺序，淘汰头部节点（最早插入）
- **LFU**：使用频率计数 + 频率桶（`LinkedHashSet`）实现 O(1) 淘汰最少使用的 key

## ⚙️ 核心技术亮点（60秒）

**[展示 02-request-lifecycle.html]**

### 亮点一：策略模式 + 模板方法模式的组合运用

缓存引擎 `LocalCache` 不关心"用什么策略淘汰"，只在 put、lookup、remove 等关键时机通知策略对象。三种策略通过实现同一个 `EvictionPolicy` 接口来替换淘汰行为。

同时，LRU 和 FIFO 都继承自 `AbstractLinkedPolicy`，共享双向链表的增删逻辑，差异仅在 `onAccess()` 方法——LRU 将节点移到尾部，FIFO 则不做任何操作。这就是模板方法模式的体现。

### 亮点二：线程安全的双重保障

虽然底层存储使用了 `ConcurrentHashMap`，但 put 操作涉及"判断容量 → 清理过期 → 淘汰 → 插入 → 更新策略"这样的复合步骤，单靠 `ConcurrentHashMap` 无法保证原子性。因此额外使用 `ReentrantLock` 将整个复合操作包裹在同一个临界区内，确保缓存状态与策略内部状态始终一致。

### 亮点三：TTL 过期的双路径清理

过期数据有三条清理路径：
1. **读路径**：`lookup()` 发现条目已过期，立即剔除并返回 EXPIRED 状态
2. **写路径**：`put()` 在插入新数据前，先扫描并清理所有过期条目
3. **后台路径**：通过 `@Scheduled` 定时任务每 10 秒执行一次批量清理

这种"懒清理 + 主动清理"的组合与 Redis 的过期策略思路一致。

**[展示 03-lru-policy.html / 04-lfu-policy.html / 05-fifo-policy.html]**

### 亮点四：LFU 的 O(1) 淘汰实现

LFU 策略使用两个 Map 和一个 `minFrequency` 指针：
- `frequencies`：记录每个 key 的访问频次
- `keysByFrequency`：每个频次对应一个 `LinkedHashSet`，保证同频次下按插入顺序淘汰
- 淘汰时直接取 `minFrequency` 对应集合的第一个元素，时间复杂度为 O(1)

## 🖥️ 演示说明（30秒）

如果现场演示，操作流程如下：

1. **启动后观察预热**：应用启动时自动将商品 1、2、3 加载到缓存
2. **验证缓存命中**：连续查询商品 1，来源显示"来自缓存"，响应极快
3. **验证数据库回源**：查询未预热的商品 8，来源显示"来自数据库"，有明显延迟（600ms）
4. **触发淘汰**：连续查询不同商品直到超过容量（8 个），观察被淘汰的 key
5. **策略对比**：切换 LRU / LFU / FIFO，重复操作，对比不同策略下被淘汰的 key 不同
6. **过期重载**：等待 45 秒后再次查询，来源显示"缓存已过期，已重新加载"

通过 Dashboard 面板可以实时观察命中率、缓存条目、热点排行的变化。

**技术层面**：
- 深入理解了 LRU、LFU、FIFO 三种缓存淘汰算法的实现原理和数据结构选型

- 实践了 Java 并发编程中 `ConcurrentHashMap` + `ReentrantLock` 的配合使用

**工程层面**：
- 采用分层架构，Web 层、Service 层、Core 层职责清晰，松耦合
- 使用不可变 record 做只读快照，保证数据安全暴露

**未来改进方向**：
- 可以引入分布式缓存支持（如 Redis 做二级缓存）
- 可以添加缓存穿透防护（布隆过滤器）和缓存击穿防护（互斥锁）
- 可以支持基于权重的淘汰策略
