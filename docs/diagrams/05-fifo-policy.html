FIFO 策略数据结构与原理

HashMap + 双向链表 · 按插入顺序稳定排列 · 访问不会改变顺序

STRUCTURE
数据结构映射
FIFO 与 LRU 共用同一套链表骨架：nodeIndex 用于快速定位节点， head / tail 维护顺序，evictKey 同样从 head.next 开始淘汰。 但核心区别在于：FIFO 的顺序只由“写入时刻”决定，访问命中不会触发重排。
nodeIndex: Map> append(node) 维持入队顺序 onAccess(entry) = no-op evictKey() = 最早进入的 key
代码语义
1. put 时仍然是尾插。
2. access 时不执行 moveToTail。
3. 因为顺序稳定，所以越早插入的节点越靠近 head。
4. 容量不足时，最先插入的节点先被淘汰。
VISUAL FLOW
PRINCIPLE
写入规则
新节点始终 append 到 tail 前。这样链表天然保持“先进先出”的入队顺序。
访问规则
访问不会改变顺序，这是 FIFO 与 LRU 的最大差异。本项目中 `onAccess` 明确为空实现。
淘汰规则
容量不足时依旧调用 `evictKey`，但因为链表顺序稳定，所以 head.next 永远是最早进入缓存的元素。
一句话总结
FIFO 的本质是“顺序只看进入时间，不看访问热度”。 在你的代码里，它和 LRU 共享双向链表基础设施，但通过让 `onAccess` 保持空实现， 实现了“同样的数据结构，不同的顺序语义”。这也正好体现了当前策略抽象设计的优雅之处。
